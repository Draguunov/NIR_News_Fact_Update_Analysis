Это приложение на Streamlit, которое анализирует новости из RSS-ленты и с помощью языковой модели пытается понять, описывают ли несколько статей одно и то же событие, а также появился ли в последней статье какой-то новый факт по сравнению с предыдущими.

Как устроена программа по шагам.

Сначала подключаются библиотеки:

* os и dotenv нужны для чтения переменных окружения;
* json нужен для работы с JSON-ответом модели;
* requests используется для HTTP-запроса к LLM API;
* feedparser нужен для чтения RSS-лент;
* streamlit отвечает за веб-интерфейс.

После этого вызывается load_dotenv(), чтобы загрузить API-ключ и другие настройки из .env файла.

Дальше задается словарь RSS_SOURCES. В нем перечислены несколько источников новостей:

* TechCrunch
* Habr AI
* The Verge
* MIT Technology Review

То есть пользователь в интерфейсе может выбрать один из этих источников, и оттуда будут подтягиваться новости.

Следующий блок — настройка LLM API.
В коде видно, что автор пробовал несколько вариантов провайдеров:

* GigaChat
* DeepSeek
* Z.ai
* GPT-4o-mini

Сейчас активным оставлен вариант для Z.ai. Оттуда берутся:

* API_URL
* API_KEY
* MODEL_NAME

Если в переменных окружения ничего не задано, используются значения по умолчанию для Z.ai:

* URL: https://api.z.ai/api/paas/v4/chat/completions
* модель: GLM-4.7-Flash

После этого в интерфейс выводятся URL, начало ключа и название модели. Это сделано, видимо, для отладки, чтобы сразу видеть, что именно подключено.

Дальше идет функция load_rss_articles(rss_url, limit=5).
Она получает адрес RSS-ленты и число новостей, которые нужно взять.

Что делает эта функция:

1. Загружает RSS-ленту через feedparser.parse().
2. Проходит по первым limit записям.
3. Для каждой записи извлекает:

   * заголовок,
   * краткое описание,
   * ссылку.
4. Для каждой статьи формирует общий текст в таком виде:
   Заголовок: ...
   Описание: ...
   Ссылка: ...
5. Возвращает список словарей. В каждом словаре есть:

   * title
   * summary
   * link
   * text

То есть статья хранится и в разобранном виде, и в виде единого текста, который потом удобно отправлять модели.

Следующая важная часть — функция build_prompt(article_texts).
Она строит промпт для языковой модели.

На вход подается список текстов статей.
Дальше:

1. Каждая статья нумеруется: Статья 1, Статья 2 и так далее.
2. Все статьи склеиваются в один большой текст.
3. Формируется инструкция для модели.

Смысл инструкции такой:

* определить, описывают ли статьи одно и то же событие;
* если да, сделать краткую сводку;
* сравнить последнюю статью с предыдущими;
* определить, есть ли в последней статье новый факт;
* если есть, выписать его отдельно;
* вернуть ответ строго в JSON с полями:

  * same_event
  * summary
  * has_new_fact
  * new_fact
  * reason

То есть ваш инференс здесь — это не классификация в ML-смысле через обученную модель, а LLM-инференс по промпту: вы подаете несколько текстов новостей и просите модель сделать логический вывод о совпадении события и новизне фактов.

Дальше идет функция call_llm_api(prompt).
Это центральная функция вызова модели.

Ее логика такая:

1. Если API_URL, API_KEY или MODEL_NAME отсутствуют, тогда вместо реального запроса используется демо-режим.
2. Иначе формируются HTTP-заголовки:

   * Authorization: Bearer ...
   * Content-Type: application/json
3. Формируется payload для OpenAI-compatible API.
   В payload передается:

   * model
   * messages
   * temperature = 0.2

В messages два сообщения:

* system: “Ты аккуратный аналитик новостей. Отвечай строго в JSON.”
* user: сам сгенерированный промпт со статьями

Потом выполняется POST-запрос через requests.post(...).
Если запрос неуспешен, response.raise_for_status() вызывает ошибку.
Если все успешно, ответ преобразуется в JSON, и из него берется:
data["choices"][0]["message"]["content"]

То есть программа ожидает стандартный OpenAI-подобный формат ответа.

Есть отдельная функция demo_llm_response().
Она возвращает заранее подготовленный JSON в виде строки.
Это нужно на случай, если API не подключен.
Тогда приложение все равно можно показать и протестировать интерфейс без реального обращения к модели.

После этого идет функция parse_llm_response(raw_text).
Она нужна для разбора ответа модели.

Проблема в том, что некоторые модели возвращают JSON не просто текстом, а оборачивают его в markdown-блок вида:

```json
{...}
```

Поэтому функция:

1. Убирает пробелы по краям;
2. Если строка начинается с ```json, удаляет эту обертку;
3. Если начинается просто с ```, тоже удаляет;
4. Если заканчивается на ```, убирает и это;
5. После очистки пытается выполнить json.loads(cleaned)

Если получилось, возвращает распарсенный объект и None как ошибку.
Если не получилось, возвращает None и текст ошибки.

Это полезный защитный слой между моделью и интерфейсом.

Дальше начинается Streamlit-интерфейс.

Сначала задаются параметры страницы:

* заголовок вкладки: “Анализ новостных обновлений”
* layout="wide"

Потом отображаются:

* большой заголовок приложения;
* краткое описание, что приложение получает новости из RSS, отправляет их в LLM и определяет, относятся ли статьи к одному событию и есть ли новый факт.

В боковой панели:

1. Заголовок “Настройки”
2. Выпадающий список выбора RSS-источника
3. Слайдер количества новостей от 2 до 10
4. Блок “Режим API”

Если API настроен, показывается зеленое сообщение “API настроен”.
Если нет — желтое сообщение, что будет использоваться демо-режим.

Дальше определяется текущий RSS URL по выбранному источнику.

Когда пользователь нажимает кнопку “Загрузить новости”:

* вызывается load_rss_articles(...)
* статьи сохраняются в st.session_state["articles"]

Использование session_state здесь важно: это позволяет не потерять загруженные статьи при последующих действиях интерфейса.

Потом программа берет статьи из session_state.

Если статьи есть:

1. Показывается подзаголовок “Загруженные новости”
2. Каждая статья отображается в раскрывающемся блоке expander
3. Внутри блока показываются:

   * summary статьи
   * ссылка на источник
   * чекбокс “Использовать статью N в анализе”

По умолчанию все чекбоксы включены.
Если пользователь оставляет галочку, индекс статьи добавляется в список selected_indices.

Дальше есть проверка:

* если выбрано меньше 2 статей, приложение пишет, что нужно выбрать хотя бы 2 статьи;
* иначе становится возможен анализ.

Когда пользователь нажимает кнопку “Анализировать”:

1. Берутся тексты только выбранных статей;
2. Строится промпт через build_prompt(selected_texts);
3. В отдельном expander можно посмотреть сам промпт;
4. Появляется spinner с текстом “Отправка запроса в модель...”

Дальше в try-блоке:

1. Вызывается call_llm_api(prompt)
2. Получается сырой текст ответа модели
3. Этот ответ парсится функцией parse_llm_response(raw_answer)

После этого в интерфейсе:

* показывается подзаголовок “Ответ модели”
* сырой ответ выводится целиком через st.code(...)

Если JSON не удалось распарсить:

* показывается ошибка “Не удалось распарсить JSON: ...”

Если парсинг успешен:
интерфейс делится на две колонки.

В первой колонке показываются две метрики:

* “Одно событие” — Да или Нет
* “Новый факт” — Да или Нет

Во второй колонке показывается:

* краткое объяснение из поля reason

Ниже отдельно выводятся:

* “Сводка” — поле summary
* “Новый факт” — поле new_fact

Если в процессе вызова модели возникает исключение, например сеть недоступна или API вернул ошибку, интерфейс показывает:
“Ошибка при вызове модели: ...”

Если статьи еще не загружены, внизу отображается подсказка:
“Нажми «Загрузить новости», чтобы начать.”

Если коротко сформулировать, что делает программа на уровне смысла, то логика такая:

1. Пользователь выбирает RSS-источник и количество новостей.
2. Программа загружает несколько последних статей.
3. Пользователь отмечает, какие статьи нужно анализировать.
4. Все выбранные статьи объединяются в один промпт.
5. Языковая модель получает задачу сравнить статьи.
6. Модель возвращает структурированный JSON:

   * одно ли это событие,
   * есть ли новый факт в последней статье,
   * краткая сводка,
   * объяснение.
7. Программа отображает этот результат в удобной форме.

Как выглядит ваш инференс:

* На вход подаются несколько текстов статей из RSS.
* Последняя статья считается “новой” и сравнивается с предыдущими.
* Модель выполняет семантическое сравнение содержания.
* На выходе получается не просто текст, а структурированное решение в JSON.
* Затем JSON разбирается и визуально отображается как аналитический отчет.

Какая здесь основная идея проекта:
это прототип системы отслеживания обновлений фактов в новостях. Не просто агрегация новостей, а попытка понять:

* повторяют ли статьи одну и ту же новость,
* появилось ли в новой статье какое-то уточнение,
* можно ли автоматически выделить это уточнение.

То есть проект ближе к задаче event matching / novelty detection на базе LLM.

Что в коде сделано хорошо:

* код разбит на логические блоки;
* есть отдельные функции для RSS, промпта, API, демо-режима и парсинга;
* есть демо-режим без API;
* есть защита от markdown-обертки в ответе модели;
* используется session_state, чтобы хранить новости между действиями;
* интерфейс позволяет пользователю вручную выбрать статьи для анализа.

Какие есть слабые места или что можно улучшить:

1. Сейчас вся логика “одинаковое событие или нет” полностью зависит от качества ответа LLM.
2. Нет дополнительной валидации структуры JSON, кроме json.loads.
3. Нет обработки нестандартных форматов ответа разных API, кроме OpenAI-compatible.
4. В интерфейс выводится начало API-ключа, что удобно для отладки, но в реальном приложении это не всегда безопасно.
5. summary из RSS может содержать HTML, и он пока не очищается.
6. Не используется поле даты публикации, хотя для новостей это важно.
7. Последняя выбранная статья автоматически считается той, в которой ищется новый факт. Это зависит от порядка статей и может быть не всегда корректно.

Если нужно совсем кратко объяснить суть одной фразой, то можно сказать так:

“Я сделал Streamlit-прототип, который берет несколько новостей из RSS, отправляет их в LLM, просит определить, относятся ли они к одному событию, и выясняет, появился ли в последней статье новый факт по сравнению с предыдущими.”
